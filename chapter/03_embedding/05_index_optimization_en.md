# Section 5: Index Optimization

The previous sections introduced embeddings, vector databases, FAISS, and Milvus. At this point, we can split documents into chunks, convert them into vectors, and search them with a vector database.

However, being able to search does not automatically mean the retrieval quality is good.

In real RAG systems, common problems include:

```text
The system finds related content, but the context is not enough.
The system finds semantically similar content that does not actually answer the question.
The knowledge base is large, and irrelevant chunks interfere with retrieval.
Top-k chunks are scattered, making it hard for the LLM to form a complete answer.
```

**Index optimization** is about designing how data is split, stored, labeled, retrieved, and passed to the LLM.

## 1. What Does Index Optimization Optimize?

A RAG index is not only a vector.

A complete index may include:

| Component | Description |
| --- | --- |
| chunk text | The actual text used for embedding and retrieval. |
| embedding vector | The vector generated by the embedding model. |
| metadata | Source, page, section, type, date, permission, etc. |
| parent / child relationship | Relationship between small chunks and larger context. |
| index strategy | Pure vector search, metadata filtering, hierarchical retrieval, etc. |

Therefore, index optimization is not just tuning `chunk_size`.

It asks questions such as:

```text
What unit should be used for precise retrieval?
How much context should be passed to the LLM?
Which metadata fields must be preserved?
Should the system filter first or search the whole collection?
How can irrelevant results be reduced as data grows?
```

## 2. Why Context Expansion Is Needed

Chunk size affects retrieval quality.

If chunks are too small, retrieval may be precise but context may be insufficient:

```text
Question:
What is Milvus suitable for?

Retrieved chunk:
"Milvus is suitable for large-scale vector search."
```

The sentence is correct, but it may not provide enough background.

If chunks are too large, they contain more context but also more noise:

```text
An entire page is stored as one chunk.
Only two sentences are relevant to the question.
The rest may distract retrieval and consume LLM context window.
```

A common strategy is:

```text
Use small units for retrieval.
Use larger context for generation.
```

## 3. Sentence Window Retrieval

**Sentence Window Retrieval** is a common context expansion strategy.

The idea is:

```text
retrieve a small sentence
but pass surrounding sentences to the LLM
```

For example:

```text
sentence before
matched sentence
sentence after
```

This allows the system to retrieve precisely while still giving the LLM enough context.

### 3.1 Conceptual Flow

```text
document
-> split into sentences
-> each sentence becomes a retrievable node
-> store surrounding window in metadata
-> retrieve the best sentence
-> replace it with the window context before generation
```

### 3.2 LlamaIndex Example

The course example is:

```text
05_sentence_window_retrieval.py
```

It demonstrates how to retrieve small sentence-level nodes and then use a postprocessor to replace the retrieved node with a larger window of context.

The important component is:

```text
MetadataReplacementPostProcessor
```

It means: after retrieval, replace the node content with a value stored in metadata. In sentence window retrieval, the metadata usually stores the surrounding context window.

## 4. Structured Indexing and Metadata

Metadata is one of the most important parts of a RAG system.

Without metadata, a retrieved chunk may look like this:

```text
"Milvus supports metadata filtering."
```

With metadata, the system can know:

```text
source: chapter/03_embedding/04_Milvus.md
section: Milvus Core Concepts
page: 5
modality: text
```

Metadata helps with:

```text
source citation
filtering
debugging
permission control
multimodal retrieval
evaluation
```

### 4.1 Markdown Structure

Markdown documents already contain useful structure:

```text
# title
## section
### subsection
```

Instead of treating the whole file as plain text, a RAG system can preserve headings as metadata.

This helps the system answer questions with better source context.

## 5. Hierarchical Index and Recursive Retrieval

As knowledge bases grow, searching the entire collection directly may become noisy.

A hierarchical strategy first retrieves a high-level source or topic, then searches inside that selected source.

```text
user query
-> high-level router index
-> selected source
-> source-specific index
-> detailed chunks
```

### 5.1 Hierarchical Index

The course example is:

```text
05_hierarchical_index.py
```

It uses multiple data sources:

```text
financials: data/C3/excel/Financials.csv
product_faq: data/C3/hierarchical/product_faq.md
regional_strategy: data/C3/hierarchical/regional_strategy.md
risk_memo: data/C3/hierarchical/risk_memo.md
```

The idea is:

```text
first layer: choose relevant data sources
second layer: search inside selected data sources
```

For example, a question about Mexico discount risk may need:

```text
financials
risk_memo
```

The first layer selects these sources, then the second layer retrieves relevant details from each source.

This reduces noise because the system does not search every document equally.

### 5.2 Recursive Retrieval

The course example is:

```text
05_recursive_retrieval.py
```

It uses:

```text
data/C3/excel/music.xlsx
```

The Excel file contains multiple sheets:

```text
1950
2000
2010
```

Each sheet is treated as a separate data source.

The first layer contains summary nodes for each sheet. The second layer contains song-level retrievers for each year.

The key component is:

```text
RecursiveRetriever
```

The flow is:

```text
query
-> root retriever searches year summaries
-> finds IndexNode(index_id="music_2010")
-> RecursiveRetriever follows index_id
-> searches the music_2010 retriever
-> returns relevant song chunks
```

`IndexNode.index_id` is the bridge between the upper-level node and the lower-level retriever.

## 6. Practical Choices for Index Optimization

Different scenarios need different strategies.

| Scenario | Suggested Strategy |
| --- | --- |
| Small teaching project | Basic chunking + FAISS is enough. |
| Long document QA | Sentence window retrieval or parent-child chunks. |
| Large multi-document knowledge base | Metadata filtering + hierarchical index. |
| API / technical documents | Preserve function names, error codes, version numbers, and metadata. |
| Table data | Route to the correct data source first, then query safely. |
| Multimodal data | Preserve modality, image path, page, and caption metadata. |
| Permission-sensitive data | Apply permission filters before retrieval. |

An optimization order can be:

```text
make the data structure correct
preserve metadata
tune chunking and indexing
add hierarchical or recursive retrieval when the data grows
```

## 7. Evaluating Index Quality

Index optimization should be evaluated with a question set.

Example evaluation questions:

```text
Which document should answer this question?
What key information should the correct answer include?
Did the system retrieve the correct chunk?
Did the LLM cite the correct source?
```

Common metrics:

| Metric | Description |
| --- | --- |
| Recall@K | Whether the correct chunk appears in the top-k results. |
| Precision@K | How many retrieved chunks are actually relevant. |
| MRR | Whether the correct result appears near the top. |
| Answer quality | Whether the final answer is complete, correct, and traceable. |
| Latency | Whether query speed is acceptable. |
| Cost | Embedding, LLM token, and retrieval pipeline cost. |

If retrieval does not return the right context, a stronger LLM or a better prompt cannot reliably fix the answer.

## 8. Key Takeaways

Index optimization is not about adding every technique. It is about choosing the right retrieval structure for the data.

```text
small chunks improve precision but may lack context
large chunks provide context but may introduce noise
sentence window retrieval retrieves small units and expands context
metadata filtering narrows the search space
hierarchical indexes are useful for large multi-source knowledge bases
recursive retrieval lets upper-level nodes point to lower-level retrievers
```

A good RAG index should help the system:

```text
find the right source
find the right section
preserve enough context
exclude irrelevant content
control latency and cost
```
